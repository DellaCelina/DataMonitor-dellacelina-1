# 요구사항 정의서 (REQUIREMENTS)

## 1. 배경 및 목적

이 저장소는 "[CRA_AI] Day3_개인과제_반도체시료관리.pdf"에서 제시된 반도체 시료 관리
시스템 아이디어에서 착안했지만, 과제 제출표(제출 항목) 상 이번 산출물은
**"데이터 모니터링 Tool (Data Monitoring Tool)" PoC** 그 자체입니다.

즉, 이번 구현의 목표는 반도체 시료 관리 시스템 전체를 만드는 것이 아니라,

> JSON 파일로 저장된 데이터를, 마치 간단한 내장형(embedded) DB처럼
> 테이블(배열) 단위로 조회/검색/추가/수정/삭제할 수 있는
> **범용(general-purpose) 콘솔 관리 도구**를 C++20으로 구현하는 것입니다.

반도체 시료/발주 데이터는 이 범용 엔진이 실제로 여러 테이블을 다루는 모습을
보여주기 위한 **데모(예시) 데이터**일 뿐이며, 엔진 코어(`DataMonitorCore`)에는
반도체 도메인 지식이 전혀 포함되지 않습니다. 다른 프로젝트에서도 이 엔진을
그대로 가져가 자신만의 테이블/레코드 스키마로 재사용할 수 있어야 합니다.

## 2. 범위

### 포함됨
- JSON 값 타입 및 파서/직렬화기 (DataMonitorCore)
- 파일 기반 저장소 (원자적 쓰기 포함)
- 테이블 단위 CRUD 및 검색 인터페이스 (ITable/JsonTable)
- 여러 테이블을 관리하는 데이터베이스 계층 (IDatabase/JsonDatabase)
- **테이블 스키마 정의/수정 및 레코드 검증** (TableSchema/FieldDefinition) -
  DB의 `CREATE TABLE` / `ALTER TABLE ADD|DROP COLUMN` / 제약조건(제약)에
  해당하는 기능을 스키마로 표현하고, insert/update 시점에 이 스키마에
  맞는지 자동으로 검사
- 콘솔 UI에서 쓰기 좋은 요약/조회/검색/추가/삭제/테이블 관리/스키마 관리
  파사드 (MonitorService)
- 반도체 도메인 더미 데이터 생성기 및 콘솔 메뉴 UI (DataMonitor 데모 앱)
- 단위 테스트 (DataMonitorTests)

### 포함되지 않음
- 실제 반도체 시료/발주 업무 프로세스 전체 구현 (승인 워크플로우, 재고
  차감 로직 등 도메인 비즈니스 로직)
- 네트워크 서버, GUI, 동시성 다중 사용자 지원
- 외부 패키지 매니저(vcpkg/NuGet)를 통한 외부 라이브러리 의존성

## 3. 기능 요구사항

| ID | 요구사항 | 설명 |
|----|----------|------|
| F-1 | JSON 파싱/직렬화 | 객체, 배열, 문자열(이스케이프/유니코드 포함), 숫자, 불리언, null을 왕복(parse→dump→parse) 가능해야 함 |
| F-2 | 테이블 생성/조회 | 지정한 이름의 테이블이 없으면 새로 생성, 있으면 로드 |
| F-3 | 레코드 추가 (Create) | id 필드가 없거나 비어 있으면 자동 채번, 있으면 그대로 사용 |
| F-4 | 레코드 조회 (Read) | 전체 조회(All), id로 단건 조회(Get), 조건자 기반 조회(Find), 필드=값 검색(FindByField) |
| F-5 | 레코드 수정 (Update) | 기존 레코드에 patch 필드를 병합(merge), 지정하지 않은 필드는 보존 |
| F-6 | 레코드 삭제 (Delete) | id로 삭제, 존재하지 않으면 false 반환 |
| F-7 | 테이블 목록/요약 | 데이터 디렉터리 내 존재하는 모든 테이블과 레코드 수 요약 |
| F-8 | 테이블 삭제 | 테이블의 기반 파일까지 완전히 제거 |
| F-9 | 콘솔 UI | 테이블 목록 확인, 페이지네이션 조회, 검색, 추가, 삭제를 메뉴로 제공 (한국어 UI) |
| F-10 | 데모 데이터 시딩 | samples/orders 두 테이블에 대해 최초 실행 시에만 더미 데이터 생성 |
| F-11 | 테이블 추가/삭제 | 콘솔/서비스 계층에서 명시적으로 테이블을 생성(`CreateTable`)하거나, 데이터 파일까지 완전히 제거(`DropTable`)할 수 있어야 함 |
| F-12 | 스키마 정의/수정 | 테이블마다 필드 목록(타입, 필수 여부, 범위/길이/허용값 제약)을 담은 스키마를 정의(`DefineSchema`)하거나, 필드 단위로 추가(`AddSchemaField`)/삭제(`RemoveSchemaField`)할 수 있어야 함 (스키마는 테이블별로 선택 사항이며, 정의하지 않으면 기존처럼 자유 형식 레코드를 허용) |
| F-13 | 스키마 기반 레코드 검증 | 스키마가 정의된 테이블에 레코드를 추가/수정할 때, 타입 불일치·필수 필드 누락·숫자 범위·문자열 길이·허용값(enum) 위반을 검사하여 위반 시 삽입/수정을 거부하고 위반 사유 목록을 반환해야 함 |

## 4. 비기능 요구사항

| ID | 요구사항 | 설명 |
|----|----------|------|
| N-1 | 영속성 | 모든 데이터는 사람이 읽을 수 있는 JSON 파일로 디스크에 저장 |
| N-2 | 원자적 쓰기 | 저장 시 임시 파일에 먼저 쓰고 rename하여, 쓰는 도중 프로세스가 죽어도 기존 파일이 손상되지 않아야 함 |
| N-3 | 외부 의존성 없음 | vcpkg/NuGet 등 패키지 매니저 없이 MSVC 표준 라이브러리만으로 빌드 가능 |
| N-4 | 재사용성 | DataMonitorCore는 특정 도메인(반도체 등)을 전혀 알지 못하는 범용 라이브러리여야 함 |
| N-5 | 테스트 가능성 | 핵심 로직(Json/JsonTable/JsonDatabase/MonitorService)에 대한 단위 테스트 존재, 반복 실행 가능(재실행 시 이전 실행의 흔적이 남지 않음) |
| N-6 | 빌드 환경 | Visual Studio 2026 (v18), PlatformToolset v145, C++20 (`stdcpp20`), Debug/Release × x86/x64 |

## 5. 산출물 매핑 (과제 제출표 기준)

이번 저장소가 제공하는 것은 과제 제출표의 "데이터 모니터링 Tool" 항목입니다.
PDF에서 설명하는 반도체 시료 발주/재고 관리 기능은 이 범용 엔진을 사용하는
"예시 소비자(consumer)"로서 `DataMonitor` 데모 앱의 `DemoSeeder`에만
등장하며, 실제 업무 시스템 구현으로 오인되어서는 안 됩니다.
